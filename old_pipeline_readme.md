
# Old Pipeline: Automatic Topic Discovery and Clustering

## Overview

Before adopting the chapter-based segmentation approach, I initially designed a pipeline that **automatically discovers topics from the podcast transcript**.

The goal of this pipeline was to convert a **long conversational transcript into structured knowledge** by:

- Extracting factual statements from the conversation
- Organizing those facts into topics
- Merging related topics
- Converting the structured information into readable text

This approach attempted to reconstruct the structure of the discussion automatically, **without relying on external metadata such as YouTube chapter timestamps**.

The pipeline uses a combination of:

- Speech-to-text models  
- Large Language Models (LLMs)  
- Sentence embedding models  
- Clustering algorithms  

---

# Pipeline Architecture

The pipeline consists of the following stages:

```
Podcast Video
      ↓
Download Audio
      ↓
Convert to WAV
      ↓
Speech-to-text Transcription
      ↓
Transcript Chunking
      ↓
Bullet Point Extraction
      ↓
Topic Generation
      ↓
Topic Mapping
      ↓
Semantic Topic Clustering
      ↓
Cluster Bullet Aggregation
      ↓
Paragraph Generation
      ↓
Section Compilation
```

Each stage progressively converts **unstructured speech → structured written knowledge**.

---

# 1. Audio Conversion

The podcast audio is first converted into a format suitable for speech recognition.

The conversion script processes `.webm` audio files and converts them into **16kHz mono WAV files**, which are commonly required by speech recognition models.

### Key processing steps include:

- Converting audio format to WAV  
- Resampling audio to **16kHz**  
- Converting to **mono channel**  
- Encoding audio as **16-bit PCM**

This preprocessing ensures that the audio format matches the input expectations of the transcription model.

---

# 2. Speech-to-Text Transcription

Once the audio is prepared, the pipeline generates transcripts using **OpenAI Whisper**.

The transcription stage converts the spoken podcast into raw text that can be processed further.

### The transcription model used is:

**Whisper (base model)**

Whisper is a **transformer-based speech recognition model** trained on large-scale multilingual speech datasets.

The transcription process outputs the transcript in structured form containing:

- text segments  
- timestamps  
- metadata  

However, for this pipeline only the **text content** is required.

The transcript is stored in:

```
data/transcripts/raw/
```

---

# 3. Transcript Chunking

Podcast transcripts are typically **very long**, often exceeding the context limits of language models.

To address this, the transcript is split into **smaller chunks of approximately 1200 words**.

### Chunking allows:

- efficient processing by language models  
- better topic identification  
- reduced memory usage  

Each transcript chunk is stored as a separate file:

```
data/transcripts/chunks/
```

### Example structure

```
chunk_001.txt
chunk_002.txt
chunk_003.txt
```

Each chunk is processed **independently** in later stages.

---

# 4. Factual Bullet Extraction

The next stage extracts **atomic factual statements** from each transcript chunk.

This step uses a **Large Language Model** running locally via **Ollama**.

### Model used

**Qwen 2.5 (7B parameters)**

The model receives the transcript chunk and is prompted to convert the conversation into **short factual bullet points**.

### Strict prompt rules ensure that the model:

- extracts only explicit information  
- produces one fact per bullet  
- avoids summarization  
- ignores conversational filler  
- preserves numbers and names exactly  

### Example transformation

Transcript sentence:

```
"I run Meesho."
```

Extracted bullet:

```
• Vidit Aatrey runs Meesho.
```

This step converts **noisy conversational speech into structured factual data**.

The extracted bullets are stored in:

```
data/bullets/
```

---

# 5. Topic Generation

After extracting factual bullets, the next stage organizes them into **logical thematic topics**.

This stage again uses the **Qwen 2.5 7B model**.

The model is prompted to:

- group related bullets together  
- assign a short descriptive title to each group  
- preserve the original bullet text  

### Example output format

```
Startup Growth

• bullet  
• bullet  

Funding

• bullet  
• bullet
```

The output represents an **initial semantic grouping** of the facts discussed in the podcast.

This output is stored in:

```
data/topics/
```

---

# 6. Topic Map Construction

The topic grouping text generated by the model is then parsed into a structured **topic-to-bullets mapping**.

This step converts the textual representation into **JSON format**:

```json
{
  "Startup Growth": [bullet1, bullet2],
  "Funding": [bullet3, bullet4]
}
```

This structured representation allows later stages to **process topics programmatically**.

These topic maps are stored in:

```
data/topic_maps/
```

---

# 7. Semantic Topic Clustering

Although the language model generates topics, **similar topics may still appear multiple times**.

Example:

```
AI startups
AI companies
AI development
```

To merge similar topics, the pipeline performs **semantic clustering using sentence embeddings**.

### Embedding model used

**SentenceTransformers — all-MiniLM-L6-v2**

This model converts topic titles into **vector embeddings** that capture semantic similarity.

### Clustering algorithm

**Agglomerative Hierarchical Clustering**

The clustering algorithm groups topics whose embeddings fall within a **predefined similarity threshold**.

The result is a set of **topic clusters**, where each cluster represents a broader conceptual theme.

Cluster outputs are stored in:

```
data/topic_clusters/
```

---

# 8. Cluster Bullet Aggregation

Once topic clusters are created, the pipeline aggregates **all bullets belonging to the topics in each cluster**.

This produces a collection of **cluster-level bullet files**, each containing the facts associated with a specific conceptual theme.

### Example structure

```
cluster_1.txt
cluster_2.txt
cluster_3.txt
```

These cluster files are stored in:

```
data/final_clusters/
```

---

# 9. Paragraph Generation

The cluster bullet points are then converted into **readable prose**.

This stage again uses the **Qwen 2.5 7B model**.

The prompt ensures that:

- each bullet is converted into a sentence  
- factual information is preserved  
- no additional knowledge is introduced  

The result is **clear explanatory paragraphs generated directly from the extracted facts**.

These paragraphs are stored in:

```
data/sections/
```

---

# 10. Chapter Compilation

The final stage combines all generated paragraphs into a **single chapter file**.

Each cluster paragraph is appended sequentially to form the final output.

The compiled chapter is saved as a **Markdown file**:

```
chapters/episode_name.md
```

This file represents the **final book-style version of the podcast episode**.

---

# Limitations of the Pipeline

While the pipeline successfully extracted **structured knowledge from conversational transcripts**, it had several limitations.

- The pipeline was **computationally expensive**, requiring multiple passes through language models.  
- **Topic boundaries were sometimes inconsistent**, since the system relied entirely on semantic grouping rather than explicit structure.  
- The pipeline contained **many intermediate stages**, making it relatively complex.

---

# Key Insight Leading to Pipeline Redesign

During development, I realized that the YouTube videos already contained **manually defined chapter timestamps**.

These chapters effectively represent **topic boundaries created by the content creators**.

Using these timestamps allowed the transcript to be segmented **more reliably** and eliminated the need for complex topic clustering.

This insight led to the development of a **simpler chapter-based pipeline**, which replaced the automatic topic discovery approach.
